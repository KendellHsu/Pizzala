# Pizzala — 美術組 / 軟體組驗收清單

> 用法:每項做完由**製作者以外的另一人**實測打勾(交叉驗收)。
> 「測法」欄寫的是最快的驗證方式;軟體組測試預設用 **Quest Link 連 PC 按 Play**。
> 相關文件:[SETUP.md](../SETUP.md)(專案設定)/ [PREFABS.md](PREFABS.md)(Prefab 組裝)。

---

# 第一部分:美術組驗收清單

## A0. 全體資產通用規範(每個資產交付前先過這關)

- [ ] 檔案放在 `Assets/Art/` 對應子資料夾(`Pizza/`、`NPC/`、`scene/`、`UI/`、`Splats/`)
- [ ] 命名全英文、無空格,格式 `類型_名稱_變體`(例:`tex_face_angry`、`splat_sauce_03`)
- [ ] 貼圖尺寸為 2 的次方(256 / 512 / 1024 / 2048),單張不超過 2048
- [ ] 模型匯入後**沒有粉紅色材質**(URP 材質遺失),縮放為真實世界尺寸(Scale Factor 調對,不是在場景裡硬拉)
- [ ] 模型 pivot 在合理位置(披薩=圓心、客人=腳底中心)
- [ ] 提交前自己先拖進場景看過一次,Console 無匯入警告紅字
- [ ] 已 push 上 GitHub 且檔案完整(.meta 一起進版控,別人 pull 下來不會 missing reference)

## A1. 披薩模型 + 三種口味貼圖

**規格**:直徑約 0.3m 的圓盤模型;三張口味貼圖(瑪格麗特/臘腸/夏威夷);面數 ≤ 2,000 tris。

- [ ] 三種口味在 **2m 外、VR 頭盔內**一眼可分辨(測法:三顆排一排戴頭盔看)
- [ ] 俯視與側視都認得出是披薩(玩家常低頭抓取)
- [ ] 背面(餅皮底)有處理,不是破面或純黑
- [ ] 貼圖 ≤ 1024,三張色調協調但主色差異明顯(紅醬/深紅臘腸/黃鳳梨)
- [ ] 面數 ≤ 2,000 tris(場上同時會有 10+ 顆)

## A2. 醬汁濺灑貼圖(髒污 decal 用)× 4~6 張

**規格**:1024×1024 PNG、透明背景、紅醬為主可混 1~2 張起司色。

- [ ] 至少 4 張,形狀彼此明顯不同(濺開/滴流/塊狀/噴點)
- [ ] 邊緣是自然噴濺,**不是規則圓形或方形**,無白邊/黑邊(alpha 邊緣乾淨)
- [ ] 中心不透明度夠高,貼在深色地板上仍看得清
- [ ] 旋轉 90°/180° 後不會看出明顯方向性(程式會隨機旋轉)
- [ ] 測法:交給軟體組做成 Splat Prefab 後,實際丟 10 顆看牆上地上效果,「看起來像同一種醬但每塊都不同」

## A3. 客人模型(或看板)+ 四張表情貼圖

**規格**:人形模型 ≤ 5,000 tris 或平面看板皆可;表情四張:正常/開心/生氣/沾醬髒臉;臉部必須是**獨立材質的 Renderer**(程式要換貼圖)。

- [ ] 四張表情同一張臉、同一 UV 佈局,換圖時五官位置不跳動(測法:Inspector 手動輪流換 mainTexture)
- [ ] 「沾醬髒臉」= 生氣或哭 + 明顯醬汁,**2m 外可辨識髒了**——這是實驗組照片素材的靈魂,值得多迭代
- [ ] 表情在 512 貼圖下仍清楚(截圖只有 512 解析度)
- [ ] 臉部 Renderer 與身體分離,層級命名清楚(`Face`)
- [ ] 若做多個客人外觀:臉部結構一致,四張表情可套用到每個變體
- [ ] 模型正面朝 +Z(FaceAnchor 截圖角度依賴這個)

## A4. 口味圖示 × 3

**規格**:256×256 PNG 透明背景,懸浮在客人頭上顯示。

- [ ] 與 A1 的三種披薩外觀**一一對應**,玩家不需說明就能配對(測法:找一個沒參與設計的人指認)
- [ ] 3m 外、VR 內可辨識(圖示實際顯示尺寸約 0.3m)
- [ ] 白色/淺色描邊或底框,在深色與淺色背景前都看得清

## A5. 玩家髒臉圖 × 2~3(髒度層次)

**規格**:1024×1024,虛擬廚師臉(或團隊自訂形象)+ 由輕到重的醬汁髒度;被砸 1 次用第 1 張、2 次用第 2 張,以此類推。

- [ ] 髒度層次遞增明顯,擺在一起一眼看出「越來越慘」
- [ ] 風格與 A3 客人髒臉一致(結算照片牆會並排出現)
- [ ] 構圖為大頭照(照片牆單格只有約 180×180 顯示)
- [ ] 交付時提醒軟體組:匯入設定要勾 **Read/Write Enabled**(程式要 EncodeToPNG)

## A6. 全螢幕醬汁濺灑圖(被砸臉特效)× 1

**規格**:1920×1080 以上 PNG 透明背景,糊在視野上的效果。

- [ ] **畫面正中央保留可視區**(中央 1/3 遮擋 ≤ 50%),玩家被砸後仍能繼續遊玩
- [ ] 邊緣濃、中央淡的分佈,像真的糊到臉上
- [ ] 戴頭盔實測 3 秒淡出過程**不引起不適**(這條由軟體組協助測)

## A7. 拍立得相框圖 × 1

- [ ] 相框內留正方形照片區(照片區 180×180 於 200×240 框內偏上)
- [ ] 下緣白邊夠寬,之後可疊手寫感文字
- [ ] 白框在結算畫面背景色上看得出邊界(不是白疊白)

## A8. 餐廳場景

**規格**:小餐廳一間,3~5 個客人位置分佈左/中/右;總面數 ≤ 100k tris(Quest 3 獨立運行預算)。

- [ ] 客人位置分佈覆蓋左/中/右三個方位,離玩家 2.5~4m(對應「各方位成功次數」數據)
- [ ] 玩家所在中島 / 出餐區有檯面放三座出餐台(高度 0.9~1.1m)
- [ ] 牆面、地板、桌面**都有 Collider**(披薩要能砸中留髒污,不能穿出場外)
- [ ] 玩家 2m × 2m 活動範圍內無視覺障礙物(會撞到的錯覺物件)
- [ ] 場景色調偏亮、偏乾淨——**髒污貼上去要顯眼**,這是實驗視覺的前提
- [ ] 有一個角落高處視野可俯瞰全場(給環境總覽截圖的 OverviewPoint)
- [ ] 戴頭盔實測:比例真實(桌子不會巨大化)、無明顯破面
- [ ] Quest Link 下維持 72fps(Stats 面板 Batches < 150 為佳)

## A9. 結算畫面版面設計(對照組 + 實驗組)★ 實驗操弄本體

- [ ] 兩版設計稿都有:對照組(數據面板)、實驗組(照片牆 + 標題)
- [ ] **兩版美術投入相當**:同字型、同配色、同背景質感——差異只能在「內容形式」,不能一版明顯比較精緻(這是實驗效度的關鍵,請組內互審)
- [ ] 數據面板字級在 VR 內 2m 距離可讀(實測,不要只在螢幕上看)
- [ ] 照片牆可容納 8~12 張照片不爆版
- [ ] 交付格式:直接在 Unity Canvas 上排,或給軟體組標注間距字級的設計圖

# 第二部分:軟體組驗收清單

## B0. 專案與建置環境

- [ ] 專案在乾淨 clone 後可直接開啟,無 missing package / 編譯錯誤
- [ ] Android platform、IL2CPP、ARM64、Min API 29、ASTC 全部設定完成(照 SETUP.md 第 1 節)
- [ ] OpenXR:Meta Quest Support 勾選、Oculus Touch Controller Profile 已加入
- [ ] Quest Link 按 Play 可直接進入 VR,頭手追蹤正常
- [ ] Build APK 一次成功,裝上 Quest 3 脫線可玩
- [ ] 脫線運行幀率穩定 72fps(OVR Metrics 或體感無抖動)
- [ ] 連續遊玩 10 分鐘不閃退、不發熱降頻到卡頓

## B1. 抓取與丟擲

- [ ] 兩手都能抓披薩,抓取點自然(不會抓到披薩瞬間翻轉)
- [ ] 放手後飛行方向與揮動方向一致,無明顯延遲感
- [ ] 丟擲距離合理:正常揮臂可丟到 4m 外的客人,輕拋 1~2m
- [ ] 用力丟牆不穿牆、丟地板不穿地(Continuous Dynamic 驗證)
- [ ] 披薩放在檯面上不滑動、不抖動
- [ ] 空中的披薩可以被另一手接住再重丟(XRI 預設行為,確認沒壞)

## B2. 手勢分類(ThrowClassifier)

測法:一人刻意做指定動作各丟 10 次,另一人看 JSON 或 Console 對答案。

- [ ] 反手(手臂橫過胸前揮出):分類正確 ≥ 8/10
- [ ] 正手(同側側臂甩出):≥ 8/10
- [ ] 過頭砸(高舉過頭往下丟):≥ 8/10
- [ ] 低手拋(低於腰部向上拋):≥ 8/10
- [ ] 左手測一輪,`hand` 欄位記錄正確、反手/正手判定不鏡像錯亂
- [ ] 每筆 ThrowRecord 的 `features` 七個原始特徵值都非零合理(releaseSpeed 2~15 m/s 區間、角度在 ±90° 內)
- [ ] 調整 ThrowTuning 閾值後,行為立即改變、不需改程式碼

## B3. 命中判定與客人反應

- [ ] 正確口味丟進 HandZone:記 `Hit`、客人開心、圖示消失、披薩被收走
- [ ] 錯誤口味丟進 HandZone:記 `WrongFlavor`、客人生氣、觸發丟回
- [ ] 砸中 FaceZone:記 `MissFace`、客人變髒臉**且維持到回合結束**、photos/ 多一張臉部特寫、觸發丟回
- [ ] 砸中 BodyZone:記 `MissBody`、身上生髒污、客人生氣(不丟回)
- [ ] 砸中環境:記 `MissEnvironment`、落點生髒污
- [ ] 沒點餐的客人被塞披薩:記 `MissBody`,不算命中
- [ ] 訂單超時:客人生氣、`missedOrders` +1、觸發丟回
- [ ] 同時多張訂單並行時,方位歸屬正確(打偏歸給最近的待餐客人)
- [ ] `reactionTime` 數值合理(0.5~15 秒間,不會出現負數或 0)

## B4. 髒污系統

- [ ] 每次失誤落點都生成髒污,位置貼合表面、無漂浮或陷入
- [ ] 無 z-fighting 閃爍(有就調 `surfaceOffset`)
- [ ] 髒污隨機旋轉/縮放/圖案,同一面牆連中 5 次看起來自然
- [ ] `DirtCount` 與肉眼計數一致,回合開始時歸零
- [ ] 50 塊髒污同場存在,幀率不掉(壓力測試:故意亂丟 3 分鐘)

## B5. 丟回與閃避

- [ ] 預警閃紅明顯可見,時長 = `telegraphSeconds`
- [ ] 披薩鎖定發射瞬間頭部位置、**不追蹤**:預警時側跨一步必躲過
- [ ] 站著不動必被砸中:醬汁糊視野 → 1.2 秒後 2 秒淡出
- [ ] 躲過的披薩砸在身後留髒污,1.5 秒後消失
- [ ] DodgeRecord 欄位驗證:左閃記 Left、右閃記 Right、下蹲記 Duck、反應時間 0.2~2 秒合理值
- [ ] 連續兩個客人同時丟回,兩發都正常結算、特效不打架
- [ ] **保險絲測試**:`enableThrowback` 關閉後整個機制完全不觸發,結算畫面閃避列自動消失

## B6. 截圖系統

- [ ] 砸臉瞬間 photos/ 出現 `face_*.png`,內容是該客人臉部特寫、構圖置中
- [ ] 截圖瞬間無可感知卡頓
- [ ] 回合結束出現 `env_*.png`,俯瞰構圖看得到大部分髒污
- [ ] 玩家被砸 N 次後出現 `player_*.png`,髒度層次對應中彈次數
- [ ] 場景中僅一個啟用相機、一個 Audio Listener(Console 無警告)

## B7. 數據記錄與匯出

- [ ] 回合結束 sessions/ 出現 `session_<時間>_<組別>.json`
- [ ] JSON 完整性抽查:每筆 throw 有 throwType、outcome、features、targetSector;photoPath 指向存在的檔案
- [ ] summary 統計與逐筆記錄一致(手算一輪準度對照)
- [ ] 結算畫面顯示的每個數字都能在 JSON 找到相同值
- [ ] `adb pull` 或 SideQuest 可取回 JSON 與照片,照片可正常開啟
- [ ] 連跑兩輪(重啟 app),兩個 session 檔案獨立、不互相覆蓋

## B8. 活動量追蹤(ActivityTracker)

- [ ] 原地站 30 秒:移動距離 < 2m(頭部微晃不該灌水)
- [ ] 刻意走動 + 下蹲 3 次 + 原地轉一圈:squatCount = 3(±0)、轉身 ≥ 300°、距離合理
- [ ] 開場站直校準有效:蹲著開場再站起來,不會誤記下蹲

## B9. 結算畫面(實驗操弄)

- [ ] `condition = Control`:只出現數據面板,所有欄位有值、無 "0/0" 或空行殘留
- [ ] `condition = Experimental`:只出現照片牆,照片=臉照+玩家照+環境照的總和,標題數字正確
- [ ] 沒有任何失誤的完美局:實驗組照片牆優雅處理(至少有環境總覽照,不是空白)
- [ ] 2m 距離頭盔內文字清晰、照片不糊
- [ ] 照片 12 張時 Grid 正常換行不爆版

## B10. 實驗操作流程(demo 現場 SOP)

- [ ] 換受試者流程 ≤ 1 分鐘:改 `participantId` + `condition` → 重啟 → 站直待命
- [ ] 兩組受試者的**遊玩過程完全相同**(髒污、變臉、丟回兩組都有),僅結算畫面不同——請刻意各玩一輪確認無其他差異
- [ ] Guardian 邊界 2m × 2m 設置完成,側跳下蹲不出界
- [ ] 中途出錯的重測流程確認:直接重啟 app 即可,舊數據不受污染
